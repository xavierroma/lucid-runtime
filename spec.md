# V1 Spec: Realtime Action-Conditioned World Model Runtime (1 GPU, 1 Worker)

## 0. Scope
V1 targets a single GPU machine running a single worker process. A separate CPU coordinator exposes a public API and assigns sessions to that worker. No Redis. No failover. If the worker dies, the session ends.

### In scope
- Create/end sessions via coordinator API
- Worker joins LiveKit room, publishes video track
- Client sends action commands via LiveKit data messages
- Worker runs inference loop and streams frames in realtime

### Out of scope
- Multiple workers / autoscaling
- Redis / durable state
- Session migration or mid-session failover
- Queueing / admission control beyond “busy → 409”
- Egress/recording pipelines
- Multi-track outputs (audio, depth, segmentation, etc.)

---

## 1. Components

### 1.1 Coordinator (Rust, CPU-only, stable address)
**Responsibilities**
- Public API for session lifecycle
- Authenticate clients
- Mint LiveKit access tokens (client + worker)
- Track session state in memory
- Assign the single worker to at most one active session

**Non-goals**
- Persistence across restarts
- High availability / multiple coordinators

### 1.2 GPU Worker (Python, 1 process, 1 GPU)
**Responsibilities**
- Connect to coordinator and register
- Wait for assignment
- Join LiveKit room as a bot participant
- Publish a video track (`main_video`)
- Receive action commands via LiveKit data messages
- Run inference loop and stream frames to the track
- Cleanup and return to idle on session end or error

### 1.3 LiveKit (Cloud)
**Responsibilities**
- Room signaling, SFU transport
- Data message delivery between clients and worker
- Video distribution to clients

---

## 2. Concepts & IDs

- `session_id`: UUIDv4 generated by coordinator
- `room_name`: deterministic from `session_id`, e.g. `wm-{session_id}`
- `worker_id`: fixed string, e.g. `wm-worker-1`
- `video_track_name`: fixed string `main_video`
- `control_topic`: fixed string `wm.control.v1` (optional if SDK/topic support varies)

---

## 3. State Machines

### 3.1 Coordinator session states
`CREATED` → `ASSIGNED` → `RUNNING` → `ENDED`  
Terminal error can be represented as `ENDED` with `error_code` set, or `FAILED` as a separate terminal state (optional).

### 3.2 Worker states
`IDLE` → `BUSY` → `IDLE`

### 3.3 Rules
- At most 1 active session at a time (`ASSIGNED` or `RUNNING`)
- Creating a session while worker is busy returns `409 Conflict`
- Coordinator and worker keep state in memory; no recovery after restart

---

## 4. Coordinator Public API (HTTPS + JSON)
Authentication: V1 may use a single shared API key.
- Header: `Authorization: Bearer <API_KEY>`

- Create session `POST /v1/sessions`
- End session `POST /v1/sessions/{session_id}:end` Idempotent.
- Get session `GET /v1/sessions/{session_id}`
- Health `GET /healthz → 200 OK`
